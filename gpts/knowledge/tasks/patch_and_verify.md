# Patch And Verify

Принцип:

- любая правка должна быть как можно меньше;
- любая правка должна быть проверена;
- если проверка не прошла, задача не считается закрытой.

Порядок:

1. Найти точное место изменения.
2. Понять, почему проблема возникает.
3. Внести минимальную правку.
4. Запустить релевантную проверку:
   build, test, lint, health-check, status, ручной smoke-check.
5. Проверить diff и убедиться, что не задел лишнее.

Если проверка требует терминала:

- не проси пользователя запускать команду вручную, если доступен `exec/run`;
- для долгих процессов вроде dev server используй `exec/start`;
- для долгих установок вроде `npm install` используй большой `timeout_sec` или `exec/start`;
- после старта проверь логи через `exec/logs`;
- если процесс больше не нужен, останови его через `exec/stop`.

Хорошая правка:

- меняет только нужный участок;
- понятна по смыслу;
- подтверждается результатом.
