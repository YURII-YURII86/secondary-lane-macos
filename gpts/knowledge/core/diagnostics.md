# Diagnostics

Когда агент попадает в незнакомый проект, он должен ответить на простые вопросы:

1. Что это за проект.
2. На каком он стеке.
3. Где его точка входа.
4. Как он запускается.
5. Как он проверяется.
6. Как он, вероятно, разворачивается.

Практический порядок:

1. Вызвать `capabilities`.
2. Вызвать `providers`.
3. Вызвать `inspect_project`.
4. Если работа будет длинной, сразу вызвать `ensureProjectMemory`.
5. Если bootstrap memory-read вернул transport/read ошибку, продолжать по живым файлам и пометить memory read как blocked, а не притворяться, что память успешно прочитана.
6. Использовать snapshot project memory до дальнейшего анализа, только если он реально получен без ошибки.
7. Если нужно быстро понять структуру, вызвать `list_dir`.
8. Прочитать ключевые файлы:
   `package.json`, `pyproject.toml`, `requirements.txt`, `Dockerfile`, `docker-compose.yml`, `README.md`, `Makefile`, `vercel.json`, `next.config.*`, `vite.config.*`, `.env.example`.
9. Определить команды:
   run, build, test, lint.
10. Определить риски:
   секреты, миграции, продовые конфиги, опасные скрипты.
11. Если в конце работы нужно показать пользователю готовую папку или запустить allowlisted приложение на локальной машине, используй доступные GPT-facing actions `open_path` и `open_app`, а не проси искать вручную.

Минимальный результат диагностики:

- тип проекта;
- основные папки;
- стек;
- команды запуска и проверки;
- важные конфиги;
- понятная структура верхнего уровня;
- понятный следующий шаг.

Минимальный результат серьёзной работы по проекту:

- project memory создана или прочитана;
- записан session record;
- если были изменения, записан change record;
- следующий агент понимает, с чего продолжать без чата.
